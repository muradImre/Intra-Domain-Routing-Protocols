We tested our Distance Vector implementation at three different levels:

1. Individual components (packet construction, timing, data structures)
2. Feature integration (how neighbor discovery + routing updates work together)
3. Full system tests (complete topologies with failures and recovery)

We started by making sure the basic stuff worked (PING/PONG, single link) and
then gradually tested more complex scenarios (multi-hop, failures, rerouting).


PACKET FORMATS

We verified that our packets follow the correct format from the spec:
- All packets have 8-byte header (type, reserved, size, source, dest)
- PING/PONG add 4-byte timestamp payload = 12 bytes total
- DV packets have variable number of (dest, cost) pairs after header
- Used htons/ntohs for all multi-byte fields (network byte order)

We manually checked the packet construction code to make sure we're using
memcpy correctly and packing fields in the right order.

TIMING CHECKS

The spec has very specific timing requirements. We tested:
- PING every 10 seconds: checked output, saw PING at 0, 10000, 20000, 30000...
- Link timeout at 15 seconds: stopped receiving PONGs, link marked dead at 15s
- Periodic DV updates at 30 seconds: updates at 0, 30000, 60000, 90000...
- Route timeout at 45 seconds: route expires if no update for 45s

We parsed the simulator output timestamps to verify these intervals.

NEIGHBOR DISCOVERY

Basic test: two nodes, one link
- Node sends PING with timestamp
- Neighbor responds with PONG (same timestamp, swapped src/dst)
- Original node calculates RTT from timestamp difference
- Neighbor info stored (router ID, RTT, port number)

We verified by looking at nb_router_num, nb_round_trip, and nb_secure arrays
getting updated correctly when PONG arrives.

ROUTING TABLE UPDATES

Checked that routes are added/removed correctly:
- Initial route to self (cost 0)
- Direct neighbor route added when PONG received
- Multi-hop routes learned from DV updates
- Routes invalidated when neighbor dies
- Routes removed when they timeout


Test Cases


SIMPLETEST1 - Basic Two-Node Test

Topology: 1 - 2 (10ms link)

This is the simplest possible test. Two nodes directly connected.
Tests:
- PING/PONG exchanges every 10 seconds
- DV updates every 30 seconds
- DATA packets going both directions (1->2 and 2->1)

Expected: All packets should be delivered successfully. Route convergence
happens almost immediately since they're direct neighbors.

How to run:
  ./Simulator simpletest1 DV > out1.txt
  diff out1.txt simpletest1.out

The output should match the provided .out file (or be very close).

SIMPLETEST2 - Link Failure Test

Topology:
    1
   / \
  2   3

Node 1 has two neighbors. The test makes link (1,2) fail and then recover.

Events:
- 100s: Send DATA from 1 to 2 (works, direct link)
- 125s: Send DATA from 1 to 3 (works, direct link)
- 150s: Link (1,2) dies
- 250s: Send DATA from 1 to 3 (still works, link (1,3) is fine)
- 300s: Link (1,2) comes back up
- 350s: Send DATA from 1 to 2 (works again)

Tests:
- Multi-neighbor discovery
- Triggered update when link fails
- Triggered update when link recovers
- Route invalidation and restoration

This one's important because it tests failure handling. When the link dies,
we should see triggered DV updates being sent.


More Tests We Created


We made 5 additional test cases to cover scenarios not in simpletest1/2:

TEST_MULTIHOP - Multi-Hop Routing

Topology: 1 -- 2 -- 3 -- 4 (linear chain)
Link costs: (1,2)=10ms, (2,3)=20ms, (3,4)=15ms

Purpose: Make sure routing works across multiple hops.

Node 1 needs to learn about node 4 even though they're not directly connected.
The route should be: 1 -> 2 -> 3 -> 4 with total cost 10+20+15 = 45ms

We send DATA from 1 to 4 and verify it takes the correct path through all
intermediate nodes. We also test the reverse direction (4 to 1).

Convergence should take ~60 seconds (need 2 rounds of DV updates to propagate
routes across 3 hops).

TEST_DIAMOND - Multiple Path Selection

Topology:
    2
   / \
  1   4
   \ /
    3

Links: (1,2)=10ms, (1,3)=40ms, (2,4)=10ms, (3,4)=10ms

Purpose: Test that Bellman-Ford picks the shortest path when multiple
paths exist.

Node 1 can reach node 4 via two paths:
- Path A: 1 -> 2 -> 4 (cost 20ms)
- Path B: 1 -> 3 -> 4 (cost 50ms)

Our implementation should choose Path A because it's cheaper. We verify by
checking that DATA packets actually go through node 2, not node 3.

TEST_REROUTE - Automatic Rerouting

Same diamond topology as above, but we break the primary path and see if
the router switches to the backup path.

Events:
- 100s: DATA 1->4 (should use path 1->2->4)
- 150s: Link (2,4) fails
- 200s: DATA 1->4 (should now use path 1->3->4)
- 300s: Link (2,4) recovers
- 350s: DATA 1->4 (should switch back to 1->2->4)

This tests:
- Triggered updates on failure
- Automatic fallback to backup route
- Triggered updates on recovery
- Return to optimal route when available

TEST_TIMEOUT - Route Expiration

Topology: 1 -- 2 -- 3 (linear)

Events:
- 50s: Link (2,3) dies
- 100s: Try to send DATA from 1 to 3

Node 1 initially learns about node 3 through node 2. When link (2,3) dies,
node 2 stops advertising the route to 3. Node 1's route should expire after
45 seconds without updates.

At t=100s (50 seconds after link died, >45s since last update), the route
should be gone and DATA should fail to deliver.

TEST_PERIODIC - Periodic Update Timing

Topology: 1 -- 2 (simple, stable network)

Events: None (just let it run for 200 seconds)

Purpose: Verify that periodic updates happen every 30 seconds even when
the topology is completely stable (no failures, no changes).

We should see DV packets at t=0, 30000, 60000, 90000, 120000, 150000, 180000.
The intervals should be exactly 30000ms.

This confirms that periodic updates are independent from triggered updates
(spec requirement).


How We Actually Tested


For each test:

1. Wrote the config file with topology and events
2. Ran: ./Simulator <test> DV > <test>_output.txt
3. Checked the output manually for correctness:
   - grep "PING\|PONG" to verify neighbor discovery
   - grep "DV" to check routing updates
   - grep "DATA" to trace packet paths
   - Look at timestamps to verify timing requirements
4. For provided tests, compared with expected output using diff

Example debugging commands we used:
  # See all PING/PONG exchanges
  grep "PING\|PONG" output.txt | head -30

  # See when DV updates are sent
  grep "Event_Xmit_Pkt_On_Link.*DV" output.txt

  # Trace a DATA packet's path
  grep "Event_Xmit_Data_Pkt\|Event_Recv_Pkt_On_Node.*DATA" output.txt

POISON REVERSE VERIFICATION

This was tricky to verify. We checked the code (lines 212-217, 286-291, etc.)
to make sure that when we advertise routes to a neighbor, we report INFINITY_COST
for any route we learned FROM that neighbor.

Example: If node 1 learned route to node 3 via node 2 (port 0), then when
node 1 sends DV update to port 0, it should advertise (dest=3, cost=65535).

We manually inspected some DV packet contents in the output to verify this,
but it's hard to fully test without more detailed packet logging.

BELLMAN-FORD CORRECTNESS

We verified the route selection logic (lines 667-701):
- New cost = advertised cost + RTT to neighbor
- Accept route if: (1) no route exists, (2) lower cost, OR (3) update from
  current next-hop with different cost
- Reject route if cost would be >= INFINITY_COST

Tested with diamond topology where there are multiple paths and made sure
we pick the minimum cost one.


Integration Testing


NEIGHBOR DISCOVERY + ROUTING

These two features work together closely. When we get a PONG:
1. Update neighbor info (RTT, router ID)
2. Update routing table (direct neighbor route)
3. If route changed, send triggered DV update

We tested this by watching a network converge from scratch. Initially no one
knows anyone. After first PING/PONG round, direct neighbors discovered. After
first DV exchange, everyone knows everyone.

FAILURE DETECTION + REROUTING

When a link dies:
1. Stop receiving PONGs
2. After 15 seconds, mark neighbor as dead
3. Invalidate all routes through that neighbor
4. Send triggered updates to remaining neighbors
5. Other routers update their routes

Tested with simpletest2 and test_reroute. Made sure triggered updates happen
within 1 second of detecting the failure.


Things We Couldn't Test Thoroughly


- Really large networks (>10 nodes) - would take too long to verify manually
- Packet loss (prob > 0.0) - spec doesn't require handling this
- Link-State protocol - not implemented
- Simultaneous link failures - didn't have time to create test case
- Link cost changes (changedelay) - didn't create a test for this
- Very rapid topology changes (links flapping) - might cause issues?


Memory Leak Testing


We used valgrind to check for memory leaks:
  valgrind --leak-check=full ./Simulator simpletest1 DV

Made sure to:
- free() all received packets after processing (recv function)
- free() alarm data when handled
- NOT free() packets after sending (system owns them after send())

Initially had some leaks from forgetting to free packets, but fixed them.
Current implementation should be leak-free based on valgrind output.


Regression Testing


Whenever we changed the code, we re-ran simpletest1 and simpletest2 to make
sure we didn't break anything. Just did:
  ./Simulator simpletest1 DV > new_out.txt
  diff new_out.txt simpletest1.out

If diff showed differences, we investigated whether it was a bug or just
minor timing differences (which are sometimes OK).
